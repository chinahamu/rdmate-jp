# バックアップ・復元手順

RDMate JPは研究データ本体を保持せず、DMP、研究課題、Dataset台帳、メタデータ、保存場所の参照情報、関連成果物、監査ログなどの管理情報をバックアップ対象にします。

## 1. バックアップ対象

| 対象 | 内容 |
| --- | --- |
| SQLite DB | ローカル利用・小規模運用の既定DB |
| PostgreSQL DB | 学内サーバ運用のDB |
| 完全JSON | 研究課題単位の管理情報スナップショット |
| エクスポート履歴 | 誰が、いつ、何を出力したかの記録 |
| `.env` | 接続先や運用モード。秘密値を含むため保護必須 |

実データ本体、個人情報を含む原データ、解析済みファイルは、RDMate JPのバックアップとは別に、各保存場所の方針で管理してください。

## 2. SQLiteバックアップ

ローカル利用や小規模な研究室利用ではSQLiteファイルを複製してバックアップします。アプリ停止中、または書き込みが少ない時間帯に実行してください。

```bash
pnpm db:backup:sqlite
```

推奨手順:

1. 現在の利用者にメンテナンス時間を告知します。
2. アプリケーションを停止、または書き込みが発生しない状態にします。
3. `pnpm db:backup:sqlite` を実行します。
4. 出力ファイル名、サイズ、作成時刻、ハッシュを記録します。
5. バックアップファイルを別ディスクまたは別サーバへコピーします。

## 3. SQLite復元

復元前には現在のDBファイルを退避してください。

```bash
pnpm db:restore:sqlite -- ./backups/rdmate.sqlite.backup
```

復元後の確認:

- アプリが起動する。
- 研究課題件数が想定と一致する。
- Dataset台帳、保存場所、メタデータが閲覧できる。
- 監査ログとエクスポート履歴が保持されている。
- `/api/health` が正常応答する。

## 4. PostgreSQLバックアップ

学内サーバ運用ではPostgreSQLの標準ツールを利用します。

```bash
pg_dump "$DATABASE_URL" > backups/rdmate-$(date +%Y%m%d-%H%M%S).sql
```

Docker ComposeのPostgreSQL service内で実行する場合の例:

```bash
docker compose exec postgres pg_dump -U "$POSTGRES_USER" "$POSTGRES_DB" > backups/rdmate-postgres.sql
```

確認事項:

- DB接続先が本番DBであること。
- バックアップファイルの保存先に十分な容量があること。
- バックアップファイルの権限が管理者のみに制限されていること。
- 暗号化またはアクセス制限された保管場所へ転送していること。

## 5. PostgreSQL復元

復元前に現行DBを退避し、利用者の書き込みを停止します。

```bash
psql "$DATABASE_URL" < backups/rdmate-postgres.sql
```

Docker ComposeのPostgreSQL service内で実行する場合の例:

```bash
cat backups/rdmate-postgres.sql | docker compose exec -T postgres psql -U "$POSTGRES_USER" "$POSTGRES_DB"
```

復元後は、アプリ起動、研究課題件数、Dataset件数、監査ログ件数、エクスポート履歴、権限設定を確認します。

## 6. プロジェクト単位JSON

研究課題単位の移行・確認には完全JSONエクスポートを利用します。JSONエクスポートは実データ本体ではなく、管理情報のみを対象にします。

出力前の確認:

- 研究課題ID
- Dataset件数
- メタデータ件数
- 保存場所件数
- 公開判断情報
- 関連成果物件数
- 監査ログ件数

取り込み前の確認:

- `schemaVersion` が対応範囲内である。
- 既存研究課題とのID衝突がない。
- 公開区分と非公開理由が保持されている。
- 参照先URLが現在も有効である。

## 7. 世代管理

推奨する最小構成:

- 日次バックアップ: 7世代
- 週次バックアップ: 4世代
- 月次バックアップ: 12世代
- リリース前バックアップ: 永続保管または長期保管

バックアップは取得するだけでなく、定期的に復元リハーサルを行います。

## 8. 論理削除

削除操作は即時物理削除ではなく、確認文字列を要求する論理削除を基本とします。依頼者、依頼日時、対象エンティティ、確認結果、処理日時を監査ログとして残します。

物理削除が必要な場合は、次を確認してください。

- 法令・契約・倫理審査上の保存義務がない。
- 関連Dataset、DMP、公開判断、関連成果物への影響を確認した。
- 削除前バックアップを取得した。
- 削除理由、承認者、実施者を記録した。

## 9. 障害時の優先順位

1. 書き込み停止と現状保全
2. 直近バックアップの所在確認
3. 影響範囲の確認
4. 復元先環境でのリハーサル
5. 本番復元
6. 復元後の整合性確認
7. 利用者への報告

## 10. 管理者チェックリスト

- [ ] バックアップ対象と対象外を利用者に周知している。
- [ ] 実データ本体は別系統でバックアップしている。
- [ ] SQLiteまたはPostgreSQLの復元手順を確認済みである。
- [ ] バックアップファイルの権限を制限している。
- [ ] 復元リハーサル日と結果を記録している。
